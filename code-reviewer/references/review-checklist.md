# 審查面向

| 優先 | 面向 | 要問的事 | 何時回報 |
|---|---|---|---|
| A | 需求與契約 | 實作是否仍符合對外承諾、API、事件、schema、權限語意 | 契約被破壞，或語意改變卻沒有清楚揭露 |
| A | 影響面 | 這個改動會波及哪些模組、服務、背景工作或共用元件 | 只修局部，卻把系統性風險留在其他邊界 |
| A | 資料與狀態 | 建立、更新、刪除、migration、快取、重試是否一致 | 可能造成遺失、重複、不一致或舊新狀態混雜 |
| A | 失敗與恢復 | timeout、partial failure、rollback、補償、重試後是否仍安全 | 壞情況不可控，或失敗路徑與成功路徑混淆 |
| A | 併發與順序 | 非同步、事件順序、重入、鎖、idempotency 是否安全 | 在負載、重試或競態下會出錯 |
| A | 安全與隔離 | 權限、租戶隔離、信任邊界、敏感資料流向是否變弱 | 造成越權、資料外洩、隔離失效或審計缺口 |
| A | 效能與擴展 | 熱路徑、N+1、扇出、批次大小、快取策略是否合理 | 在真實流量或資料量下有退化風險 |
| A | 可觀測與運維 | log、metrics、trace、feature flag、rollback、告警是否足夠 | 出事時難定位、難回退、難判斷影響範圍 |
| B | 註解與實作 | 註解、docstring、文件是否描述真實行為 | 會誤導維護者、呼叫端或測試撰寫者 |
| B | 命名與功能 | 類別、方法、變數、布林語意是否準確 | 名稱過度承諾、方向相反或掩蓋副作用 |
| B | 測試訊號 | 測試是否真的攔得住這次改動的主要風險 | 高風險改動缺乏足夠訊號 |

- 小問題若共享同一個 root cause，合併成一列。
- 沒有足夠證據時，標示為 `風險`，不要寫成已證實缺陷。
